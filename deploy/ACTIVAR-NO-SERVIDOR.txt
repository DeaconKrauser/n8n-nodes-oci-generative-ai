Ativar o nó OCI no n8n (servidor ~/n8n)
======================================

1) Confirmar o NOME do serviço no compose (no servidor):

   cd ~/n8n
   docker compose -f docker-compose-main.yml config --services

   Se aparecer "n8n_main", o EXAMPLE já usa essa chave em "services:".
   Se aparecer outro nome, edita docker-compose.oci-merge-main.yml e troca a chave.

   Caminho no contentor: use /home/node/.n8n/custom/n8n-nodes-oci-generative-ai
   (não /opt/...) para evitar "Unrecognized node type: CUSTOM.*" ao executar.

   Volume de dados (OBRIGATÓRIO para community nodes + queue): o merge EXAMPLE
   inclui /home/ubuntu/n8n/data -> /home/node/.n8n no main e no worker (mesmo bind).
   Sem isto, nodes/node_modules não existe e aparece Unrecognized em Evolution/Chatwoot.
   No host: mkdir -p /home/ubuntu/n8n/data && sudo chown -R 1000:1000 /home/ubuntu/n8n/data
   Se ainda tens ficheiros úteis só dentro do contentor (antes do bind), faz backup
   com docker cp ANTES do próximo up. A partir do PC de dev: scripts/push-compose-merge-and-up.sh

2) Copiar o exemplo para ficheiros reais (se ainda não existirem), a partir de ~/n8n:

   cp custom-extensions/n8n-nodes-oci-generative-ai/deploy/docker-compose.oci-merge-main.EXAMPLE.yml \
      docker-compose.oci-merge-main.yml

   Se EXECUTIONS_MODE=queue (como no teu caso), o ficheiro de merge do *worker*
   NÃO é criado automaticamente: tens de o copiar tu para a raiz do projeto n8n.

   cp custom-extensions/n8n-nodes-oci-generative-ai/deploy/docker-compose.oci-merge-worker.EXAMPLE.yml \
      docker-compose.oci-merge-worker.yml

   ATENÇÃO: em "services:" a chave tem de ser o NOME REAL do serviço (ex.: n8n_worker),
   exactamente como em "docker compose ... config --services". NUNCA uses "n8n:"
   como atalho: o Compose interpreta como um serviço NOVO sem image/build e falha
   com "service \"n8n\" has neither an image nor a build context".

   Depois descobre o nome exacto do *serviço* no worker (não confundir com o nome
   do contentor Docker):

   docker compose -f docker-compose-worker.yml config --services

   Abre docker-compose.oci-merge-worker.yml e em "services:" troca a chave
   "n8n_worker" pelo nome que o comando acima imprimiu (ex.: worker, n8n-worker).
   O volume host em "volumes:" deve continuar a apontar para a pasta real onde
   está o pacote (ex.: /home/ubuntu/n8n/custom-extensions/...).

   Se N8N_CUSTOM_EXTENSIONS tiver vários caminhos com ";", coloque o path do OCI
   por último (bug: só o último pacote CUSTOM é carregado).

3) Validar YAML (não reinicia nada):

   docker compose -f docker-compose-main.yml -f docker-compose.oci-merge-main.yml config > /dev/null && echo OK

   Se o teu docker-compose-main.yml incluir ou referenciar o worker, o comando acima
   pode falhar com "n8n_worker has neither an image" — nesse caso valida com os
   quatro -f (main + merge-main + worker + merge-worker).

   OBRIGATÓRIO: no output resolvido, o serviço n8n_main tem de listar
   N8N_CUSTOM_EXTENSIONS (senão o nó OCI não aparece na UI nem nas execuções):

   docker compose -f docker-compose-main.yml -f docker-compose.oci-merge-main.yml \
     -f docker-compose-worker.yml -f docker-compose.oci-merge-worker.yml \
     config | sed -n '/^  n8n_main:/,/^  [a-zA-Z0-9_-]\+:/p' | grep N8N_CUSTOM_EXTENSIONS

   Se não imprimir nada, abre docker-compose.oci-merge-main.yml: a chave em
   "services:" tem de ser n8n_main (ou o nome real do serviço), NUNCA só "n8n:"
   sem image no compose principal.

   Com queue, validar também o worker. Se aparecer
   "service \"n8n\" has neither an image nor a build context", o teu
   docker-compose-worker.yml quase de certeza define um serviço "n8n" incompleto
   (fragmento para juntar ao main) ou inclui ficheiros que deixam "n8n" sem image.
   Diagnóstico rápido:

   grep -nE '^  n8n:|\bn8n:\s*$' docker-compose-worker.yml

   Correcções típicas: (a) apagar ou completar o bloco "n8n:" no worker.yml;
   (b) renomear esse bloco para o mesmo nome do serviço principal (ex.: n8n_main:)
   se for só overrides do editor; (c) validar/subir com *três* ficheiros, se o
   teu fluxo for sempre main+worker juntos:

   docker compose -f docker-compose-main.yml -f docker-compose-worker.yml -f docker-compose.oci-merge-worker.yml config > /dev/null && echo OK

   Se com só worker+merge já passa o "config":

   docker compose -f docker-compose-worker.yml -f docker-compose.oci-merge-worker.yml config > /dev/null && echo OK

4) Aplicar (recria contentores com volume + env novos):

   docker compose -f docker-compose-main.yml -f docker-compose.oci-merge-main.yml up -d

   Com queue, subir o worker com o merge OCI (usa a mesma combinação de -f que
   passou no "config" do passo 3):

   docker compose -f docker-compose-worker.yml -f docker-compose.oci-merge-worker.yml up -d

   ou, se só funcionou com main+worker+merge:

   docker compose -f docker-compose-main.yml -f docker-compose-worker.yml -f docker-compose.oci-merge-worker.yml up -d

   Nome do contentor (para docker exec):  docker compose ps
   (com os mesmos -f que usaste no up; o nome na coluna NAME pode ser tipo
   n8n-n8n_worker-1, não "n8n_worker".)

5) No UI n8n (porta 5678), procurar "Oracle OCI" / modelo de chat OCI.

Reverter: voltar a subir só com o ficheiro original (sem -f do merge).
